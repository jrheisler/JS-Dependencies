Overview

Nurox Nexus is an interactive visualization tool for exploring codebase dependencies. It turns a JSON dataset—typically produced by the Nurox Dependency Analyzer—into a live, force-directed graph that reveals relationships between modules, files, and packages.

The interface uses a responsive, three‑column layout in dark mode. A collapsible sidebar latch lets you maximise the canvas while keeping controls a click away.

Layout & Navigation

  • Left Sidebar — Search, Filters, Physics controls, and Legend. Collapse or expand it using the header button or floating latch.
  • Center Canvas — Interactive graph with live HUD overlay and highlighting metrics.
  • Right Sidebar — Node details, insights, contextual help, and quick documentation access.
  • Shared Header — Global controls, status messaging, color modes, and export actions.

  • Type to find a node by name or path with incremental feedback.
  • Enter jumps to the next match, centering the node and highlighting neighbors.

Filters

  • Show disconnected only — Focus on files unreachable in every profile.
  • Show dynamic-only — Surface nodes that rely solely on dynamic evidence.
  • Show deferred-only — Highlight modules loaded lazily.
  • Hide externals — Exclude packages from node_modules.
  • Hide dynamic edges — Remove dynamic/heuristic edges.
  • Minimum degree — Slider with live count to prune low‑degree nodes.

Physics Controls

  • Charge — Node repulsion strength.
  • Link distance — Ideal spacing of connected nodes.
  • Link strength — Spring stiffness of edges.

Legend

The legend now reflects profile-aware statuses. Each badge lists a color and explanation for the active profile.

  • Reachable (current profile) — Active profile entrypoints can reach the node; the default bright green.
  • Reachable in other profile(s) — Only reachable when you switch profiles; shown in aqua.
  • Deferred-only — Only discovered through deferred or lazy edges; highlighted in violet.
  • Dynamic-only — Backed solely by dynamic loading, coverage, or keep rules; shown in amber.
  • Test-only / Build-time only — Loaded exclusively through the corresponding edge phases.
  • Disconnected (all profiles) — Never reached in any profile; bright pink and the primary deletion candidates.
  • Switching to Type, Degree, or Package color modes still recolors nodes dynamically based on that perspective.

Central Visualization

  • D3.js force‑directed graph with zoom and pan.
  • Drag a node to pin; double‑click to unpin.
  • Nodes update color based on the selected color mode; dynamic edges are dashed.
  • Pin positions persist locally, so custom layouts survive reloads.

HUD & Highlighting

  • The floating HUD shows live counts for nodes, edges, reachable, disconnected, and dynamic-only nodes after filters are applied.
  • Click any metric pill to spotlight the corresponding nodes or edges directly in the graph.
  • Metric highlights stack with manual selections and respect the current color mode.

Dependency Logic Explained

The Nurox Dependency Analyzer scans every source file in your project and builds a visual map of how your code connects. The reclassification system now produces richer statuses that factor in multiple profiles and edge phases.

1. Adding Files as Nodes

Every project file becomes a node in the graph. External packages such as those inside node_modules are tracked separately so you can hide them with a single toggle.

2. Tracking Connections

Whenever one file imports or exports from another, Nurox records that link as an edge. Each edge carries metadata about how it loads (static, deferred, or dynamic) and which phase it belongs to (runtime, test, or build).

3. Profiles & Entrypoints

Nurox identifies entry nodes by reading hints such as main, module, or exports fields and can honour multiple profiles (for example, app vs. tests). Switching profiles in the header recalculates reachability from that profile’s entrypoints.

4. Reachability Analysis

For each profile the analyzer performs several traversals: full reachability, without deferred edges, without dynamic edges, runtime-only, test-allowed, and build-allowed. These sets decide whether a node is reachable now, only via deferred edges, only via dynamic signals, and which execution phase touches it.

5. Classifying Nodes

The results of those traversals are merged into statuses. Nodes reached in the active profile are tagged Reachable (current profile). Nodes hit only in other profiles are labeled Reachable in other profile(s). Files that require deferred edges become Deferred-only; those backed solely by dynamic evidence become Dynamic-only. If the only paths are through test or build edges, they receive Test-only or Build-time only. Anything untouched in every profile falls into Disconnected (all profiles), marking it as a safe deletion candidate.

Summary

StatusMeaning
Reachable (current profile)Reached from entrypoints in the active profile.
Reachable in other profile(s)Not reached right now but reached when another profile is active.

This profile-aware model mirrors real execution paths for each scenario, letting you distinguish between truly disconnected code and modules that are simply deferred, dynamic, or tied to a different profile.

Desktop Viewer Flow

This section summarises how the packaged desktop app orchestrates the HTML frontend and the language-specific crawlers that generate dependency graphs.

1. Desktop Controller (NuroxView.exe)

The native wrapper is compiled from lib/nurox_view.dart. When launched it binds an HTTP server on a random open port, loads any bundled sample graph, generates a bearer token, and opens the default browser to the local viewer URL with the token attached.

Key responsibilities:

  • Serving the viewer UI. Requests to /, /nuros_nexus.html, or /nexus.html return the viewer HTML with the token injected into a <meta name="nurox-token"> tag so the browser can authenticate later API calls.
  • Static assets. Other non-API GET requests are treated as static asset lookups, resolved against the working directory, the script directory, and packaged public/ folders.
  • Graph storage. The controller keeps an in-memory graph that merges nodes and edges from crawlers, deduplicating them while preserving the richest metadata.
  • API surface.
    • GET /api/languages lists discoverable crawler executables on disk together with availability metadata.
    • POST /api/crawl accepts a project root plus one or more language identifiers, runs the matching crawler executables, loads their default *Dependencies.json outputs, and merges them into the active graph.
    • GET /api/graph returns the merged dependency graph for the frontend to render.

Authentication is enforced on every /api/* route: the viewer must send the bearer token that was injected into the HTML or provided via the ?token= query parameter.

2. Primary Frontend (nuros_nexus.html)

public/nuros_nexus.html hosts the D3-based graph canvas, sidebar controls, and modal help system. A bootstrap script adds a <base> tag so assets work whether the file is opened directly or served through the embedded web server.

When loaded inside the desktop viewer, the page reads the injected token or URL parameter and stores it as an Authorization header for API calls. In this “viewer mode” it enables the Local Crawl panel, letting users:

  1. Fetch the list of available crawlers via GET /api/languages, showing each language with availability state and executable path metadata.
  2. Start a crawl by posting { root, languages, clear } to /api/crawl, displaying progress and reporting merged node/edge counts when the controller responds.
  3. Reload the current merged graph from /api/graph, immediately feeding it into the layout and filtering pipeline.

Outside of viewer mode the UI still works for static JSON files (drag-and-drop or manual load) but the crawl controls remain disabled, guiding users to launch the desktop app.

3. Legacy Shim (nexus.html)

public/nexus.html exists for backward compatibility. It immediately redirects to nuros_nexus.html while displaying a brief message, allowing older shortcuts or bookmarks to keep working with the renamed viewer.

4. Language Crawlers

The controller knows about eight language identifiers, each mapped to a set of executable names and a default output file. It probes the working directory, the script or packaged public/ folders, and the system PATH until it finds a match. The expected JSON filenames are jsDependencies.json, pyDependencies.json, goDependencies.json, rustDependencies.json, javaDependencies.json, kotlinDependencies.json, csharpDependencies.json, and dartDependencies.json respectively.

Each crawler is implemented as a standalone Dart CLI in lib/ and can be compiled to a native executable. They all follow the same broad pattern: recursively scan source files for the target language, infer imports/modules, compute reachability from entry points, and write the dependency graph in the schema that the viewer understands. The top-of-file documentation summarises what each crawler inspects:

  • JavaScript / TypeScript (lib/jsDependency.dart): Extracts ES module imports, resolves them to local files or external package nodes, estimates LOC, and labels nodes as used, unused, or side-effect-only based on reachability analysis.
  • Python (lib/pyDependency.dart): Scans .py files, evaluates import/from statements, interprets __main__ guards and packaging metadata as entry points, resolves modules to files or pip: externals, and marks usage state from reachability.
  • Go (lib/goDependency.dart): Walks .go files, reads go.mod, resolves imports within the module or to std:/go: externals, and treats package main files with func main() as entry points.
  • Rust (lib/rustDependency.dart): Parses mod, use, and extern crate directives, consults Cargo.toml for binaries and dependencies, and resolves module paths to files or crate: externals.
  • Java (lib/javaDependency.dart): Collects packages, imports, and classes with public static void main, resolving imports to internal classes or Maven-style external identifiers.
  • Kotlin (lib/kotlinDependency.dart): Reads package declarations and imports, infers declarations and fun main entry points, and expands wildcard imports to internal files when possible.
  • C# (lib/csharpDependency.dart): Analyses namespaces, using directives, and Main methods to connect files and identify external references such as dotnet: or nuget: packages.
  • Dart (lib/dartDependency.dart): Tracks import, export, and part directives, resolves package: URIs relative to the current project or as pub: externals, and detects main() entry points in common locations.

During a crawl the controller runs each selected language sequentially, reading the emitted JSON from the project root. Missing outputs or non-zero exit codes are logged but do not stop the overall process; any successful outputs are merged into the shared graph and exposed back to the frontend.

Export & Persistence

  • SBOM CSV exports node metadata (state, type, package, LOC, degree). Use Filtered only to scope the export.
  • Auto‑load attempts URL param (e.g., ?load=graph.json) or defaults to jsDependencies.json, falling back to a demo dataset with status messaging.
  • Sidebar collapse state and node pin positions are saved locally to keep your layout intact.

Help Modal

  • The right-sidebar help link opens an accessible modal containing this guide without leaving the graph.
  • The modal traps focus, supports Esc to close, and returns focus to the originating control.
  • When opened for the first time the help content loads lazily, keeping initial page weight minimal.
